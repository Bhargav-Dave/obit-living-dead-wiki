Data Processing

This module focuses on how raw obituary materials are transformed into research-ready data. A driving question for this section is: In what formats do the obituaries arrive, and what work is required to make them usable for analysis?

Across sources, obituaries may appear as page images, PDF documents, HTML snippets, or already extracted text. For each format, we document how the text is obtained, how language and layout are cleaned, and how information is structured into variables that support close reading and systematic analysis.

Input formats

Different sources deliver obituary content in different technical formats. This shapes the first steps of the processing pipeline.

  • PDFs and page images. Many archival sources (state archives, historic newspapers, and some funeral program collections) provide page-level PDFs or image files. These require conversion to machine-readable text, either through built-in text layers or through OCR.
  • Born-digital text. Some sources (e.g., Legacy.com and contemporary news sites) expose obituary text in HTML or other structured formats. Here the main task is to strip boilerplate and isolate the obituary body while preserving headings, dates, and bylines where available.
  • Embedded obituaries. In historic newspapers, obituaries may appear inside columns or blocks that also contain advertisements or non-obituary content. In these cases, we treat the page as a container and use additional preprocessing to separate obituary-like segments from surrounding text.

Each source in the Data collection module is tagged with its dominant input format so we can standardize downstream steps as much as possible.

Cleaning & modernization

Once we have access to raw text, we clean and modernize it enough to support both close reading and automated extraction, while still preserving the historical character of the language.

  • From PDF/image to text. Where no text layer is available, we run OCR to obtain a first-pass transcription. For structured PDFs, we use simpler text extraction tools but still check for broken line breaks, column artifacts, and missing characters.
  • Premodern and noisy English. Older obituaries often contain premodern spelling, irregular punctuation, and OCR noise. For this material, we explore light-touch modernization and spelling normalization, informed by work already started by Eunkyung (see the prototype tool at oldobits).
  • Page-level preprocessing. When obituaries are embedded in larger blocks of text, we experiment with segmentation strategies (e.g., splitting on headings or keywords) and simple filters to distinguish obituary-like segments from surrounding content.
  • Standard text normalization. Across sources, we standardize character encodings, fix common line-break issues, remove duplicated artifacts, and retain a parallel copy of the minimally processed text for archival purposes.

Structuring & information extraction

After basic cleaning, we focus on extracting information that can be analyzed systematically. This combines hand annotation and large language models (LLMs), grounded in the project’s qualitative coding framework (documented on the Research design page).

  • Hand-annotated training sets. Using the shared coding scheme, we create small annotated corpora that record how human coders identify key entities and relationships. These annotations serve both as standalone qualitative material and as reference examples for automated methods.
  • LLM-assisted extraction. Once we have stable coding schemes and examples, we use LLMs to support information extraction. This can include:
    • Converting obituary prose into structured variables and categorical codes,
    • Named entity recognition (people, places, organizations),
    • Identifying kin and mapping kinship ties,
    • Identifying community roles and organizational participation.
  • Relationship- and table-ready outputs. Extracted information is organized into tidy tables (e.g., one row per obituary, one row per relationship) that can support descriptive analysis and cross-source comparison (for example, kinship ties and organizational affiliations).

Outputs & linkages

Processed data from this module connects back to both the Data collection and Research design modules. For each source, we aim to maintain:

  • A record of the original files or URLs used to generate the text,
  • A cleaned text version suitable for close reading, and
  • One or more structured datasets suitable for systematic analysis.

For higher-level notes on where obituary texts originate and how coverage is defined by state, period, and source, see the Research overview and Data collection pages.
